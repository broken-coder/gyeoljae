# gyeoljae v0.2.0-rc.1 — Developer Preview

gyeoljae is a **Developer Preview**. The `v0.1.0` tag remains on its original
commit and is not moved by this release line. Interfaces, CLI flags, and state
formats may change between preview releases; deploy shadow-first and pin exact
versions. RC releases must be published with `npm publish --tag next`, never
as `latest` (see CONTRIBUTING).

**Versioning note.** The `0.1.1-rc` and `0.1.1-rc.1` release candidates were
repository-internal and never published to npm. Because this line changes
approval behavior (fail-closed authorization, proposal-cycle keys), it was
renumbered to `0.2.0-rc.1` — the first version intended for npm, under the
`next` dist-tag.

## Changes since `v0.1.0`

1. **Fail-closed approver authorization.** `validateApprovalReply`
   only ever emits `approved-candidate` when a configured authorizer
   recognizes the replying user. Bot/system messages (`bot_id`/`subtype`) are
   never approvals; a well-formed approval without a user id, without a
   configured authorizer, or from an unlisted user degrades to `needs-human`
   with a distinct reason — recorded, never silent. `gyeoljae-listen` wires
   this via `--approvers-file` (JSON array of user ids) in both live and
   fixture modes.
2. **Approvals bound to proposal identity and expiry.**
   `PendingRequest` can carry `proposal_id`/`proposal_digest`/`version`/
   `expires_at`; candidates carry the identity through for record-time
   re-checks, and a reply after `expires_at` no longer approves.
3. **Proposal-cycle keys.** The watch orchestrator keys blocked/done
   transitions and approval notifications per proposal cycle when the source
   supplies identity; a second proposal on the same item re-blocks and
   re-notifies. Record-time digest mismatch rejects stale approvals
   (`stale_rejected` in the pass summary). The **GitHub watch source now
   supplies this identity**: the latest `## Approval requested` comment's id
   and body digest. Sources without identity keep ref-only keys.
4. **Durable inbox and outbox.** `gyeoljae-listen --inbox-dir`
   stores each Socket Mode envelope durably **before** acking (a persist
   failure skips the ack so Slack redelivers) and replays
   persisted-but-unprocessed envelopes on start. The `Notifier` accepts an
   optional `Outbox` with pending→sending→sent states and stored receipts, so
   a crash between a remote send and its checkpoint is reconciled instead of
   blindly resent. Both are opt-in; without them the previous semantics
   apply. New public subpaths: `gyeoljae/slack/inbox`, `gyeoljae/notify/outbox`.
5. **Message bodies never persist.** `redacted_text` (the message body with
   only secrets masked) was already stripped from new public envelopes; the
   store now also sanitizes records written by older versions on read, and
   rewrites the whole file clean on the next upsert.
6. **No lost reads.** `conversations.history` with a known ack point and
   `conversations.replies` follow Slack cursor pagination (hard cap 100 pages —
   exceeding it aborts the run rather than truncating), so replay after an
   outage reads the complete window. GitHub issue and comment reads paginate
   past the first 100 results.
7. **Packaging & verification.** First npm-ready packaging of the library and
   CLIs: explicit exports map (root + documented subpaths incl. inbox/outbox),
   `gyeoljae-poll`/`-listen`/`-watch` bins, controlled `files` list,
   `engines.node >=22`, clean `prepack` build. Reproduction tests for every
   item above, a packed-consumer smoke (authorization paths included), a
   Docker runtime smoke in CI, and a sanitize gate that blocks credentials and
   private issue-tracker identifiers from tracked files.

## Upgrade notes (breaking for preview deployments)

- **No auto-approvals without an allowlist.** A live `gyeoljae-listen`
  without `--approvers-file` still runs but records every well-formed
  approval as `needs-human` (`authorization-not-configured`). Create a JSON
  array of approver user ids (e.g. `["U0EXAMPLE001"]`, mode `0600`) and pass
  it, or the approval loop stops auto-recording.
- **Watcher state keys changed format.** Blocked/done/approval-needed keys
  gain a proposal-cycle suffix when the source provides identity (the GitHub
  source now does). Items whose proposals are still open may re-notify and
  re-transition once after the upgrade. One-time, per open proposal.
- **Bot/subtype/userless events are no longer validated as approvals**, in
  fixtures as well as live. Dry-run fixtures must carry a `user` field for
  events that should produce candidates.
- **First poll after upgrade may read a larger window.** Replay from the last
  ack now paginates to completion instead of stopping at one page; a long
  outage window means proportionally more Slack API calls.

## Known limitations (unchanged in this preview)

- Notification delivery is **deduplicated at-least-once**; without an
  `Outbox` wired in, a crash between a remote send and its checkpoint can
  repeat a notification (the opt-in outbox reconciles that window).
- Local JSON state requires **exactly one writer per path** and is not
  written atomically ([single-writer contract](../deployment/local-json-state.md)).
- `gyeoljae-watch` is chat-shadow but **ledger-live**: marker-driven GitHub
  label/comment/close transitions execute whenever the token permits.
- GitHub reads have **no rate-limit retry/backoff**; a rejected page fails the
  whole pass, which restarts from the beginning next interval.
- The package is **ESM-only**; `require()` is not part of the contract.
- `GitHubIssuesWatcher` (label-driven read-only preview) still keys
  approval-needed events per label cycle, not per proposal cycle.
- Proposal `expires_at` is enforced when a deployment's pending file supplies
  it; the shipped watcher glue does not populate it yet.
